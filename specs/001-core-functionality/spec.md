## 用户故事 (Web版)

作为一名用户，我希望能够：
1. **上传音频文件**（最大300MB，1-2小时时长），自动进行语音转文字和说话人识别
2. **查看转写进度**，了解当前处理状态（转写中、生成中、完成、失败、已取消）
3. **编辑会议标题**，初始按时间戳自动生成，纪要完成后自动总结标题（<30字），支持手动编辑
4. **编辑说话人标识**，将"Speaker A/B/C"自定义为人名（可选，部分或全部）
5. **使用提示词模板**生成结构化会议纪要，支持预设模板和自定义自然语言提示词
6. **查看历史记录**，按时间线浏览所有会议记录和生成的纪要版本
7. **导出会议纪要**为Markdown或带编号的txt格式
8. **配置AI服务**，在Web界面设置LLM服务的API密钥并验证
9. **手动清理音频**，立即删除音频文件释放空间

## 功能性需求

### 1. 音频上传与处理

#### 1.1 上传约束
- **文件格式**: 支持常见音频格式（mp3, m4a, wav, webm, mp4等）
- **文件大小**: 最大300MB（前端校验，超限时提示）
- **时长限制**: 1-2小时（前端提示，软约束）
- **上传方式**: 单文件上传，不支持实时录音

#### 1.2 处理流程（异步）
```
上传 → 语音转文本 + 说话人识别(LLM) → 完成 → 用户生成纪要 → 自动生成标题
```

**说明**: 优先使用LLM进行语音转文本，同时完成说话人识别

#### 1.3 状态管理
会议状态枚举：
- `UPLOADING` - 上传中
- `TRANSCRIBING` - 转写中（显示进度百分比）
- `GENERATING` - 纪要生成中
- `COMPLETED` - 完成
- `FAILED` - 失败（显示错误原因）
- `CANCELLED` - 已取消（用户中途取消）

### 2. 转写结果与说话人管理

#### 2.1 转写输出
- **格式**: 时间戳 + 说话人标识 + 文本内容
- **说话人数量**: 3-10人，无硬性上限
- **默认标识**: `Speaker A`, `Speaker B`, `Speaker C`...
- **语言支持**: 中文为主，包含英文单词

#### 2.2 说话人自定义
- **时机**: 转写完成后，用户可编辑
- **操作**:
  - 点击说话人标识，输入自定义名称
  - 可部分命名，其余保留默认`Speaker X`
  - 修改后全局生效（所有历史记录同步更新）

### 3. 会议纪要生成

#### 3.1 提示词系统
**预设模板**（可扩展）：
- "客户业务描述、需求背景、痛点和期望、项目计划"
- "技术讨论要点、技术决策、行动项"
- "会议决议、行动项、讨论要点"

**用户自定义**：
- 自然语言描述
- 可保存为个人模板（多套）
- 可设置默认模板

**结构化选项**（可选补充）：
- 勾选框：会议决议、行动项、讨论要点、待办事项等

#### 3.2 生成规则
- 每次生成一个版本
- 可更换提示词重新生成，产生平行版本
- 基于转写文本（不含说话人标识）+ 用户提示词

#### 3.3 版本管理
- **展示**: 时间线形式
- **关系**: 平行版本（无父子关系）
- **操作**: 用户可删除任意版本

### 4. 会议标题管理

#### 4.1 标题生成规则
| 阶段 | 标题格式 | 示例 |
|------|---------|------|
| 初始（上传时） | 时间戳 | `2026-04-14 14:00会议` |
| 纪要完成后 | LLM自动总结 | `Q2产品规划讨论` |

#### 4.2 标题约束
- **长度**: 最大30个中文字符
- **自动生成**: 基于纪要内容，由LLM提取核心主题
- **手动编辑**: 用户可随时修改标题

#### 4.3 用户体验
- 转写完成后显示初始标题（时间戳格式）
- 首次生成纪要后，标题自动更新为简洁主题
- 用户可在任何时候手动编辑标题

### 6. 数据存储与清理

| 数据类型 | 保存策略 |
|---------|---------|
| 音频文件 | 转写完成后7天自动删除 + 支持手动删除 |
| 转写文本 | 永久保存 |
| 会议纪要 | 永久保存（所有版本） |
| 说话人映射 | 永久保存 |
| 用户模板 | 永久保存 |

**自动清理机制**：
- 服务启动时检查过期音频
- 定时任务每日清理

**手动清理**：
- 用户可在会议详情页手动删除音频文件
- 删除前需确认操作
- 删除后不影响转写文本和纪要内容

### 5. API配置与管理

#### 5.1 支持的服务
**语音转文本服务**（优先LLM）：
- 使用LLM进行语音转文本 + 说话人识别（推荐）
- 支持模型：
  - 豆包
  - 千问
  - Claude (Anthropic)
  - OpenAI GPT-4o（支持音频输入）
  - 其他主流支持音频输入的大模型（可扩展）

**纪要生成服务**（LLM）：
- 豆包
- 千问
- Claude (Anthropic)
- OpenAI GPT-4/GPT-4o
- 其他主流大模型（可扩展）

#### 5.2 配置方式
- **存储**: 前端Web配置 → 项目`config.json`（纯文本）
- **验证**: 配置时提供验证入口，显示具体错误信息
- **选择**: 用户可手动选择使用哪个服务，或设置默认服务

#### 5.3 单用户场景
- 无需登录系统
- 配置文件全局共享

### 6. 导出功能

#### 6.1 导出格式
- **Markdown**: 标准Markdown格式
- **TXT**: 带编号标题格式（`1. xxx`, `1.1 xxx`）

#### 6.2 导出内容
- **不包含说话人标识**（纪要是整理后的内容）
- **完整导出**: 一次性导出整个会议纪要

### 8. 页面结构

1. **主页** - 音频上传入口
2. **历史记录列表** - 所有会议记录概览（显示标题、时间、状态）
3. **会议详情页** - 标题编辑、转写结果、说话人命名、纪要生成、版本时间线、手动清理音频
4. **API设置页** - 服务配置与验证

## 非功能性需求

### 1. 架构设计原则
- **阶段一（LLM语音转文本+说话人）** 与 **阶段二（LLM纪要生成+标题）** 解耦
- 支持灵活切换不同LLM服务
- 遵循YAGNI原则，不实现未明确要求的功能
- LLM优先策略：优先使用支持音频输入的LLM进行一体化处理

### 2. 错误处理
- **所有错误必须显式处理**
- 使用 `raise ValueError("message") from err` 进行错误包装
- 用户可见的错误信息必须友好、可操作
- API调用失败时显示具体原因（如"密钥无效"、"额度不足"等）

### 3. 性能约束
- 异步处理，不阻塞用户操作
- 进度更新实时反馈（百分比）
- 大文件处理超时保护

### 4. 测试要求
- **TDD**: 所有功能从失败测试开始
- **参数化测试**: 优先使用`@pytest.mark.parametrize`
- **集成测试**: 优先于Mock，使用真实依赖

## 技术栈约束（宪法要求）

- **语言**: Python
- **Web框架**: 最小化依赖，优先使用Python标准库
- **测试**: pytest + 参数化测试
- **错误处理**: 显式异常处理，无全局变量
- **构建**: Makefile标准化操作

### P0 (必须有)
- 音频上传与300MB限制
- LLM语音转文本 + 说话人识别
- 基础纪要生成（提示词输入）
- 历史记录查看

### P1 (应该有)
- 说话人自定义命名
- 预设模板 + 自定义模板保存
- 多版本纪要管理
- 导出功能（Markdown/TXT）
- 会议标题自动生成与手动编辑

### P2 (可以有)
- API配置Web界面
- 进度百分比显示
- 音频自动清理 + 手动清理
